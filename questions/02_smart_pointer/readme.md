# 量化交易系统：低延迟行情分发组件设计与实现

## 1. 题目背景与核心诉求

本题目要求设计一个量化交易系统中的低延迟行情数据分发组件。系统包含多个行情源（Publisher）和策略模块（Subscriber），需满足以下条件：

* **动态生命周期管理**：策略模块可随时订阅、退订或被销毁。
* **绝对内存安全**：强制使用标准库智能指针，严禁悬挂指针访问，杜绝内存泄漏及循环引用。
* **多线程与低延迟**：支持多 I/O 线程高频推送，主线程低频修改订阅状态。读取路径（推送）需极低延迟，避免锁竞争。
* **异常安全**：支持策略在回调函数内部安全地退订自身。

## 2. 考察核心知识点

本题主要考察 C++ 现代系统级编程的以下核心能力：

1. **所有权语义与智能指针**：`shared_ptr` 与 `weak_ptr` 的图结构设计，打破循环引用。
2. **并发数据结构设计**：读写不对称场景下的 Copy-on-Write (CoW) 模式。
3. **C++ 内存模型与原子操作**：`std::atomic` 以及 Acquire-Release 内存序的精准控制。
4. **STL 高效算法**：Erase-Remove 惯用法及右值/移动语义在内存重排中的表现。

## 3. 架构设计与解法思路

针对读（极高频行情推送）写（低频订阅/退订）极度不对称的业务特征，本方案采用 **Copy-on-Write (写时复制)** 结合 **原子智能指针** 的架构：

* **生命周期解耦**：`Subscriber` 的绝对生命周期由外部主控引擎通过 `shared_ptr` 管理。`Publisher` 内部仅持有 `weak_ptr`，作为非拥有型观察者，确保策略销毁时自动失效，不引发僵尸内存。
* **无锁读取路径**：`publish` 方法中不使用任何互斥锁。通过原子加载（`load`）获取当前订阅列表的 `shared_ptr` 快照，确保在遍历回调期间，即使后台发生退订操作，当前迭代器依然绝对安全。
* **有锁写时复制**：`subscribe` 与 `unsubscribe` 使用 `std::mutex` 串行化写操作。拷贝旧列表、清理失效指针、插入新元素后，通过原子替换（`store`）更新全局指针。此举将同步开销完全转移至写端。

## 4. 关键语法与底层实现细节

本实现涉及以下 C++ 深度特性：

### 4.1 智能指针的底层机制

* **`std::enable_shared_from_this<Subscriber>`**：允许策略对象在自身的成员函数（如 `on_market_data` 回调）中，安全地获取指向自身的合法 `shared_ptr` 并发起退订操作，避免产生多个独立控制块导致的 Double Free 崩溃。
* **`wp.lock()` 的原子性**：在推送遍历时，将 `weak_ptr` 提升为 `shared_ptr` 是一次原子的控制块检查与引用计数加一操作。这严格保证了只要提升成功，在触发回调期间策略对象绝对不会被析构。

### 4.2 C++20 原子智能指针与内存序 (Memory Order)

* **`std::atomic<std::shared_ptr<T>>`**：C++20 特性（取代了旧的全局哈希锁机制），支持对包含控制块指针的 16 字节智能指针结构进行原子替换，部分硬件下可实现无锁（如 `CMPXCHG16B` 指令）。
* **Acquire-Release 语义**：
* **写端 `store(..., std::memory_order_release)**`：构建内存屏障，确保 `new_list` 内部元素的拷贝和构造操作，绝对发生在指针替换操作之前。
* **读端 `load(std::memory_order_acquire)**`：与写端的 Release 形成同步关系，保证读取到新指针时，必定能看到完整且正确的新列表内存状态，防止数据撕裂。
* **写端 `load(std::memory_order_relaxed)**`：极限性能优化。因该 `load` 操作已被 `std::mutex` 保护，互斥锁已提供了足够的 acquire 屏障同步其他写线程，故此处仅需保证加载操作的原子性即可，无需额外的同步开销。



### 4.3 Erase-Remove 惯用法

* 结合 `std::remove_if` 与 `vector::erase` 进行列表清理。底层采用双指针（读/写迭代器）算法进行 `O(N)` 级别的内存重排，利用 `std::move` 转移有效资源并覆盖将亡值，最后统一截断尾部无效内存，避免了多次 `erase` 带来的 `O(N^2)` 数据搬运开销。
